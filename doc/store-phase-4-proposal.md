# 商城第四阶段提案：用户认证、个人中心与收货地址管理

## 目标

移除固定用户 `1`，为商城建立可靠的用户身份边界。用户可以注册、登录、退出，管理自己的收货地址，并在结算时选择地址。购物车、订单和地址必须按登录用户严格隔离。

## 非目标

- 不接入支付宝、微信或其他真实支付渠道。
- 不实现找回密码、短信验证码、第三方登录和管理员角色。
- 不迁移现有固定用户数据到指定账号；首次注册用户在空用户表中通常取得 ID `1`，但业务不依赖该事实。
- 不引入 Redis Session 或分布式登录；当前单体应用使用容器 HttpSession。

## 功能需求

- 用户名注册，BCrypt 保存密码；注册后自动登录。
- 用户名和密码登录；退出后服务端 Session 失效。
- `JSESSIONID` 使用 HttpOnly Cookie；生产 HTTPS 环境可通过配置启用 Secure。
- 保留 Spring Security CSRF 防护，前端使用独立的 XSRF Token Cookie/Header。
- 当前用户接口返回用户 ID、用户名、昵称和手机号。
- 地址支持新增、编辑、删除、设为默认；每个用户最多一个默认地址。
- 删除默认地址后，自动将剩余地址中的第一条设为默认。
- 结算页只能选择当前用户地址；订单保存收货信息快照。
- 未登录访问购物车、订单、地址和个人中心 API 返回 JSON 401。
- 商品列表、分类和详情保持公开。

## 数据表

- `store_users`：用户名、BCrypt 密码摘要、昵称、手机号、启用状态和时间戳。
- `store_addresses`：用户 ID、收货人、电话、省市区、详细地址、邮编、默认标记和时间戳。

## 验收标准

- 两个账号不能读取、修改或使用对方的购物车、订单和地址。
- 注册、登录、退出以及会话恢复可用，密码字段从不返回前端。
- 未认证受保护 API 返回 401，缺失或错误 CSRF Token 的写请求被拒绝。
- 地址 CRUD、默认地址切换和结算选址可用。
- 下单只接受 `addressId`，并使用服务端读取的地址生成订单快照。
- `mvn test` 和 `npm run build` 通过，应用可连接已启动的 MySQL 运行。

## 假设

- 用户名为 3-30 位字母、数字或下划线；密码至少 8 位且最多 72 位。
- 当前前后端同源部署，Vite 开发代理转发 `/api`。
- `hibernate.ddl-auto=update` 继续负责本阶段建表；生产环境后续应改用版本化迁移工具。
