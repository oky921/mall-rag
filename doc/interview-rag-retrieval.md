# 面试口述：RAG 向量库与多路召回检索链路

> 对应简历要点：基于 Milvus 构建商品信息与图片向量库，按业务实体粒度分块保障语义完整性；设计查询预处理器（意图识别+查询改写+元数据过滤），实现语义向量、精确匹配、类别过滤三路并行召回，结合大模型生成回答。

---

## 一、面试角度分析（这部分在考察什么）

这条简历要点信息密度很高，面试官通常会沿着四个方向逐层下钻：

1. **分块策略**：为什么不按固定 token 切？不同文档类型为什么要用不同策略？"业务实体粒度"具体指什么？
2. **查询预处理**：意图识别怎么做的？规则还是模型？查询改写解决了什么问题？元数据过滤表达式长什么样？
3. **多路召回**：三路分别召回什么？为什么要并行而不是串行？召回结果怎么去重和合并？
4. **工程闭环**：Milvus 挂了怎么办？向量库和 MySQL 数据不一致怎么办？大模型生成的 Prompt 怎么约束？

核心要讲清楚的是：**这不是一个"调一下 similaritySearch 就完事"的玩具 RAG，而是一条有意图分流、有精确兜底、有业务重排、有容错降级的工程化检索链路。**

---

## 二、面试者口述回答

### 1. 背景与问题

这个项目的场景是电商智能助手，用户可能问"给我找一款 1000 元以内的降噪耳机"，也可能问"空气炸锅第一次用怎么开箱"，还可能直接报一个商品编号"P10001 这款适合什么人"。如果只做最朴素的 RAG——把所有文档丢进 Milvus，用户问什么就向量检索什么——会有三个问题：第一，商品编号是精确查询，语义检索未必能召回对的；第二，用户说"服饰鞋包"是类别意图，纯语义检索会混入不相关品类；第三，说明书、品牌故事、活动规则这三类知识结构完全不同，混在一起切分会破坏语义边界。

所以我把检索链路拆成了"入库分块 → 查询预处理 → 三路并行召回 → 重排过滤 → 大模型生成"五个阶段，每一阶段都针对具体的业务问题做了设计。

### 2. 向量库构建：两个 Collection，按业务实体粒度分块

首先，我在 Milvus 里建了两个 Collection，走的是两条不同的链路。文本知识库走 Spring AI 的 VectorStore 抽象，Collection 叫 `first_rag_docs_utf8`，1024 维，COSINE 距离度量；图片向量库走的是原生 MilvusClientV2，Collection 叫 `mall_product_images`，用的是 DashScope 的多模态 Embedding 模型 qwen3-vl-embedding，也是 COSINE，索引类型选了 AUTOINDEX。图片那边的 Schema 是我手动定义的，包含 id、product_id、sku_id、title、image_url、embedding 向量字段和一个 JSON 类型的 metadata 字段，用双重检查锁做懒加载初始化，第一次写入或查询时才建 Collection 并 load。

重点是文本这边的分块策略。我没有用通用的固定 token 切分，而是**按业务实体粒度分块**，具体来说分两类：

**商品信息**——每个商品就是一个 Document，粒度就是商品实体本身。Document ID 用 `store-product:{productCode}` 这种稳定 ID，metadata 里带 source=mysql-store、type=product、product_id=商品编码、category、price 这些字段。商品变更是通过 Outbox 模式异步同步到向量库的，用 SHA-256 做版本对账保证最终一致。

**本地 Markdown 知识库**——这部分是说明书、品牌故事、活动规则三类文档，我在 `MarkdownKnowledgeIngestService` 里**按文档类型用了三套不同的分块策略**：

- 说明书（manual）：按单个段落切，每块最多 500 字，80 字重叠。因为说明书是操作步骤型的，一个段落讲一个操作，切太大会把多个步骤混在一起，切太小又丢失上下文。
- 活动规则（promotion_rule）：也是单段落切，450 字、60 字重叠，策略名叫"原子规则"。因为促销规则之间是独立的，满减规则和退货规则不能混在一个 chunk 里，否则模型会把不同规则的条款串到一起。
- 品牌故事（brand_story）：用滑动窗口合并多个段落，900 字、120 字重叠。因为品牌故事是叙事性的，段落之间有连续性，合并起来语义更完整。

每个 chunk 除了正文，我还注入了结构化头：文档类型、文档标题、来源文件、主题标签、摘要，然后才接正文。同时 metadata 里带了 doc_type、topic、summary、chunk_index、chunk_strategy 这些字段，这些后面查询预处理会用到。长文本切分时我还做了句子边界感知——优先在句号、分号处断开，避免把一句话切成两半。

这样设计的好处是：**每个 chunk 既是语义完整的单元，又携带了足够的元数据用于精确过滤，检索时可以按 doc_type、按 category 做精准收敛。**

### 3. 查询预处理器：意图识别 + 查询改写 + 元数据过滤

查询进来后不是直接丢给 Milvus，而是先过一道 `RagQueryPreprocessService`，它会输出一个 `RagQueryPlan`，包含四样东西：识别出的意图、改写后的查询、主文档类型、一组 Milvus 过滤表达式。

**意图识别**用的是关键词加权打分。我给三类文档各配了一组领域关键词，比如说明书匹配"空气炸锅、首次、清洁、故障、白烟"这些词，活动规则匹配"满减、优惠券、预售、价保、退货"。用户查询跟每组关键词做包含匹配，累计命中数最高的就是主文档类型。如果最高分是 0，说明是通用问题，走 general 分支不加过滤。这里我故意没用大模型做意图识别，因为关键词匹配延迟低、可控、可解释，而且我们的意图空间是有限的、已知的。

在确定主文档类型后，我还会做一轮**细粒度意图检测**。比如同样是说明书，如果命中"故障、白烟、异响"就是 manual_troubleshooting，命中"安全、用电、烫伤"就是 manual_safety，否则是 manual_usage。活动规则会细分成价保、售后、预售、物流四个子意图。

**查询改写**是我觉得比较关键的一步。原始用户查询往往很短，比如"第一次用怎么办"，直接拿去做向量检索召回质量很差。所以我会根据识别出的文档类型和子意图，往查询里追加领域扩展词。比如识别到是说明书的使用意图，我会在原始查询后面拼上"澄风AF-65智能空气炸锅 使用说明书 操作步骤 故障排查 安全注意事项"这样的扩展文本。这不是替换原始查询，而是拼接——既保留了用户的原始表达，又补充了领域语义锚点，让向量检索能命中更精准的文档。

**元数据过滤表达式**是根据文档类型直接生成的 Milvus filter，比如 `source == 'local-md' && doc_type == 'manual'`。如果多个文档类型都命中了，我会生成多个过滤表达式，每个都去召回一遍，这样不会漏掉跨类型的知识。

### 4. 三路并行召回

预处理完之后进入 `RagService.searchDocuments`，这里是三路并行召回的核心：

**第一路，语义向量召回。** 用改写后的查询做 similaritySearch，topK 设到 10（MAX_TOP_K）。如果预处理器生成了过滤表达式，就按每个过滤表达式各召回一次；如果没有过滤表达式，就做一次无过滤的全库语义检索。我还会对查询做一层类别扩展——如果检测到用户提到"手机、数码"等词，会在查询后面补上"数码类商品 手机 笔记本 耳机"这样的同义词扩展，提升召回率。

**第二路，精确匹配召回。** 我用正则 `P\d{5}` 从用户查询里提取商品编号，如果提取到了，就以商品编号本身为查询、带上 `product_id == 'P10001'` 的过滤表达式去 Milvus 做一次精确召回，topK 设为 1。这一路是为了解决"用户直接报商品编号"的场景——语义检索对精确编号的召回是不可靠的，必须走元数据精确匹配。

**第三路，类别过滤召回。** 我会从查询里检测用户请求的类别，比如"数码、手机"映射到 digital，"服饰、羽绒服"映射到 clothing，然后对每个类别用 `category == 'digital'` 的过滤表达式做一次召回。这一路是为了解决"给我推荐服饰类商品"这种类别意图——纯语义检索可能混入其他品类，加类别过滤能精准收敛。

三路结果会通过 `mergeDocuments` 做去重合并，去重 key 优先用 product_id，其次用 title，最后用 content 全文。值得注意的是，第二路和第三路只在**不是本地 Markdown 查询**时才触发——也就是说如果用户问的是说明书、活动规则这类知识，就只走第一路带 doc_type 过滤的语义召回，不会触发商品精确匹配和类别过滤，避免无关商品混入知识问答。

### 5. 重排与业务优先级

合并去重之后，结果不是直接取 topK，而是先过一道 RerankModel 重排，再做两轮业务优先级排序。

RerankModel 目前是 PassThroughRerankModel 透传实现，但接口已经留好了，后续可以接真正的 rerank 模型。重排之后，我做第一轮 `prioritizeBusinessMatches`：对每个结果算一个业务匹配分——商品编号精确命中加 100 分，类别命中加 30 分，价格在预算内加 20 分，是商品类型加 5 分。这样用户问"P10001"或"500 元以内的服饰"，对应的结果会被排到最前面。

第二轮 `prioritizeQueryPlanMatches` 是针对本地知识库的：如果预处理器识别出这是本地 Markdown 查询，命中所规划 doc_type 的结果加 1000 分，确保知识问答场景下正确文档类型的片段排在最前。

最后还有一个 `isRelevant` 相关性过滤：如果结果命中了 businessMatch 标记（product_id、category_budget、local_doc_type 任一），直接放行；否则看向量距离是否小于等于 maxDistance 阈值 0.55。这个阈值是经过 Ragas 评估调出来的——一开始设太宽导致大量无关上下文，后来收到 0.55 才稳定。

### 6. 大模型生成

拿到最终的相关文档后，进入生成阶段。Prompt 是我精心构造的：先把每个文档的 metadata（标题、类型、来源、商品编号、类别、价格等）格式化进去，再拼正文，然后附上对话历史和最新问题。Prompt 里有一段明确的约束指令——"严格遵循用户指定的商品编号、类别、预算、颜色、季节、场景、功能约束"，"如果用户指定了类别，不要推荐其他类别的商品除非明确说明"。

多轮对话我也做了处理：`buildRetrievalQuery` 会把最近 4 条用户消息拼接到当前问题前面，作为检索 query，这样用户说"那有没有更便宜的"这种指代消解问题，检索时能带上上文语境。对话历史最多保留 8 轮，超出就截断最近的。

还有一类特殊处理是**范围外问题兜底**。如果检索没召回任何相关文档，我不会让模型自由发挥，而是根据问题关键词返回预设的拒答——比如命中"医疗、诊断"就回答"医疗诊断不属于当前商品知识库范围"，命中"法律、诉讼"就回答法律类不在范围内。这是为了防止模型对知识库覆盖不到的问题产生幻觉。

### 7. 容错与降级

整个链路有三级容错。第一级，单路召回失败不影响其他路——精确匹配和类别过滤那两路如果抛异常，我 catch 住只打 debug 日志，继续用语义召回的结果。第二级，整个 RAG 检索失败时，`searchDocumentsForChat` 会 catch 异常返回空列表，chat 方法检测到没有知识库上下文就降级成无知识回答。第三级，在商品推荐场景，`StoreProductRecommendationService` 调 RagService 失败会降级成纯 MySQL 词法匹配——用关键词命中分 + 是否精选 + 销量排序来兜底，保证 Milvus 挂了用户还能搜到商品。

另外商品推荐这里还做了一层**数据一致性保障**：Milvus 召回回来的 product_id 只是候选，必须回 MySQL 用 findByCodeIn 查出来、过滤掉已下架的、校验价格是否在预算内，才会返回给用户。Milvus 是最终一致的，可能有过期候选，MySQL hydrate 是强制的，所以向量库里多一个少一个商品不影响最终结果正确性。

### 8. 总结

一句话概括：我构建了一个**意图驱动的多路召回 RAG 链路**——入库时按业务实体粒度分块保证语义完整，查询时先识别意图和改写查询，再语义、精确、类别三路并行召回，重排时用业务规则做优先级调整，生成时用结构化 Prompt 约束模型行为，全链路有分级容错和 MySQL 兜底。效果上经过 Ragas 评估，context_recall 从 0.40 提升到 0.92，answer_relevancy 从 0.60 提升到 0.78。

## 三、预判追问

### 追问 1：意图识别为什么用关键词匹配而不是用大模型做分类？

关键词匹配延迟低、成本低、可解释、可调试。我们的意图空间是有限且已知的——就是说明书、品牌故事、活动规则三类加几个子意图，关键词词表是可控的。用大模型做意图分类每次查询要多一次模型调用，增加几百毫秒延迟和 token 成本，而且意图判断不可解释、不好调试。如果未来意图空间变复杂、关键词覆盖不住，可以再考虑用小模型或大模型做分类，但现阶段关键词匹配是性价比最高的选择。

### 追问 2：三路召回的结果怎么保证不重复？去重逻辑是什么？

去重是在 `mergeDocuments` 里做的，用 LinkedHashMap 按 documentKey 去重。documentKey 的优先级是：有 product_id 就用 `product:{PRODUCT_ID}`，有 title 就用 `title:{标题}`，都没有就用 `content:{全文}`。同一个商品可能被语义召回和精确匹配同时命中，但 product_id 相同，所以只保留第一次出现的。用 LinkedHashMap 是为了保留插入顺序，语义召回的结果天然排在前面。

### 追问 3：maxDistance 阈值 0.55 是怎么定的？

是 Ragas 评估迭代出来的。第一轮没设阈值或者设太宽，context_precision 只有 0.40，大量无关上下文被召回，16/30 的问题检索不到上下文。后来调到 0.55，配合元数据过滤和业务重排，context_precision 提到 0.86，无上下文行数降到 2/30。这个阈值本质是在召回率和精确率之间找平衡——太严会漏召回，太宽会引入噪声。我们更偏向精确率，因为电商场景下推错商品比少推几个代价更大。

### 追问 4：向量库和 MySQL 数据不一致怎么办？

两个层面保障。第一，写入侧用 Outbox 模式——商品变更先写 MySQL 的事务里同时写 Outbox 事件表，定时任务异步消费事件同步到 Milvus，有重试退避和 SHA-256 版本对账，保证最终一致。第二，读取侧强制 MySQL hydrate——Milvus 召回回来的 product_id 只是候选 ID，必须回 MySQL 查出来确认商品存在、没下架、价格在预算内，才会返回。所以即使向量库有过期数据，最坏情况只是多召回一个不存在的候选然后被过滤掉，不会返回错误结果给用户。

### 追问 5：RerankModel 现在是透传，为什么不做真正的重排？

接口已经设计好了，`RerankModel` 接口加 `RoutedRerankModel` 路由实现，目前接的是 `PassThroughRerankModel` 保持原始顺序。之所以先用透传，是因为当前业务重排逻辑（product_id +100、category +30、budget +20）已经能解决主要的排序问题，而且这些业务规则是确定性的、可解释的。真正接入 rerank 模型（比如DashScope 的 rerank 接口）会增加一次网络调用和延迟，在当前数据量下收益不明显。但架构上已经留好了扩展点，配置里切换 provider 就能接入，不需要改代码。
